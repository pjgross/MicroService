Peter gross : 5) Decentralised Governance

Created by Peter Gross, last modified on Sep 15, 2020


In the past we have had very set standards for the databases, languages, logging tools and platforms that you have to use.


In traditional projects there is a standard for almost everything

  • Which dev platform to use
  • Which database to use
  • How logs are created
  • and more....


In the Microservices architecture, you may have standards for providing logs, a cloud provider and security standards etc... to follow but these are just requirements for the product teams and how they implement them is up to them. 

There should still be some standards to allow the operation of the applications, but these should be provided as services in their own rights, such as centralised logging, monitoring and audit systems.  These provide a public API that the other services can call using any technology they desire.

A microservice product team are able to choose NodeJS for a REST API, or .net application for document processing, a Postgres ot MongoDB for the database etc...


We have to remember that:

  • Each product team is fully responsible for its service;
  • So they choose the technology that best suites their service
  • This is enabled by making sure the services are loosely coupled


Since the services should be decoupled, then making changes down the line to take advantage of technology changes should not be a concern of other product teams, as long as the interface to that service provides the same functionality.


Motivation

  • Enables making the optimal technology decisionbs for the specific service
