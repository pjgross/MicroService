Peter gross : 1) Meeting Microservice Guidelines

Created by Peter Gross, last modified on Sep 12, 2020


These are the design decisions i made during design that aligned to to the Microservice characteristics published by Martin Fowler


Componentisation via Services

This application is being split out into a number of services that provide REST api's if they are publishing synchronous interfaces, and use a message queue for the asynchronous interfaces.

Organised around business capabilities

The services have been chosen around business functions rather than splitting out all functions into services.

This provides a well defined interface, with the aim of keeping each business service independent of all other services (this was set as a goal at the start, but in practice the authorisation service may have been a synchronous call to the authorisation service if this had not been the case)

Products not Projects

Since I am the only user and developer of the system, this is built in.

Smart endpoints. and dumb pipes

All of the logic is inside of the services, including authorisation and routing rules.  The REST api by definition is dumb, and I have built no logic into the Messaging server.

Decentralised Governance

I have standardised the technology stack for easy of use, I made some decisions one common error messaging and express middleware, but i would always expect these to happen

Decentralised Data Management

Each service will have its own data store, which for an update and performance point of view has proven to be a refreshing change. 

In phase 2 I will implement a management information service which can be used for MI reports of all of the data across the services, which will not put load on the individual services.

The main problem i need to solve is in the setup of test data, since data needs to be consistent across all services, and how this would be accomplished on a large scale.

Infrastructure automation

I have developed GitHub workflow scripts that run the Jest unit/Integration tests when the pull request is created.

There are also GitHub workflow scripts to deploy to DigitalOcean on a successful merge into the master branch.

The development environment has been automated using skaffold, and the Jest scripts are run automatically when saving a file.

Design for failure

The Kubernetes config has been setup to monitor the application and restart any service that has stopped.

All errors are logged to the console, but to keep the first iteration time down, the implementation of the EFK stack will be left to phase 2

Evolutionary design

The whole purpose of this project is to experiment what it takes to deliver a Microservices application and operate it in production. 

The role of environment management has shown me that we need to understand the bigger picture more than ever, and if we automate and improve inter product team communications, may not be needed as a seperate role in the future, but especially in the testing side of things, we are not mature enough to reach this point yet.
