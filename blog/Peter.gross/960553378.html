Peter gross : 4) Smart endpoints. and dumb pipes

Created by Peter Gross, last modified on Sep 15, 2020


The problem with the SOA archicture was there was too much logic built into the ESB, which made them complicated and very expensive


In Microservices all of the logic is in the end points, and the communication channels are relatively dumb, and are modeled around web based protocals like REST, Graphql and GRPc.


Typically each of the services will have a public REST api that they can talk to each other over

Although we normally use other services to insulate the services so they don't know have to know exactly where the other services are located.

Typically using a service mesh for synchronous communication or a message queue for asynchronous communications.


Notes:

  • Direct connections between services is not a good idea

  • It is better to use a discovery service

  • or a gateway

Each product team concentrates on the services they have advertised on their public api, and the message they send and receive from the message bus.

This should make the development alot simpler than SOA's ESB architecture.


Motivation

  • Accelerate development
  • Make the app easier to maintain
