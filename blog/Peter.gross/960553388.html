Peter gross : 8) Design for failure

Created by Peter Gross, last modified on Sep 15, 2020


Moving away from single large monoliths to many smaller components means there are more moving parts

  • Handling network traffic becomes a bigger concern
  • There is alot that can go wrong
  • The service must assume failure can happen and handle it gracefully
  • Logging and Monitoring should be in place from the first service


This is why there is growing support for handling some of these issues for you:

Service meshs

  • DevOps gain visibility into the microservices.
  • Application-wide TLS can be applied instantly.
  • Controlled deployment (canary, blue/green) is possible with granular traffic control.
  • Detect service failures and avoid them with circuit breakers.

Message queues

  • Publish your key events and not worry about who uses them, great for service decoupling
  • If a service goes off line or a new service is added, it can receive all messages it has missed, which is great for handling service outages.

Kubernetes

  • You design a service state, and kubernetes will keep the services running to that state, automatically restarting services if they fall over
  • Decouples service communications by communicating to service wrappers, so you do not have to know where that service is located


Motivation

  • Increase the services reliability
